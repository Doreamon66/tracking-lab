# 运营活动自定义埋点规范（魔方组件化架构）

> 本规范定义运营活动（营销活动页）的埋点方案，采用**组件化架构**，与通用位置模型（SKILL.md）是完全独立的两套体系。
> 适用场景：WPS 会员营销活动页、促销落地页、福利中心等基于魔方平台搭建的运营页面。

---

## 一、整体架构：3 事件 + 组件化 event_name

运营活动只使用 **3 个固定事件**，所有埋点维度通过字段组合来区分：

| 事件名 | 含义 | 触发时机 |
|--------|------|----------|
| `show_rubikpage` | 页面加载 | 活动页首次加载完成，上报一次 `event_name=act` |
| `show_rubikwindow` | 组件/弹窗展示 | 页面内各 UI 组件进入可视区域或弹窗弹出时 |
| `click_rubikbtn` | 用户点击 | 用户点击按钮、卡片、链接等可交互元素时 |

**与位置模型的核心区别**：位置模型的事件名由业务场景动态决定；组件化模型的事件名固定为上述 3 个，业务语义完全由 `position` + `event_name` + `data1~data8` 字段组合承载。

---

## 二、公共字段结构

每次上报均包含以下字段：

| 字段 | 类型 | 必填 | 来源 | 说明 |
|------|------|------|------|------|
| application | string | 是 | 平台自动 | 应用 ID，无需业务关注 |
| url | string | 是 | 平台自动 | 当前页面 URL |
| version | string | 否 | 平台自动 | 固定 `2.0.0` |
| **position** | string | 是 | 运营配置 | 楼层编号，标识组件在页面中的位置 |
| **event_name** | string | 是 | 运营配置 | 事件标识，标识组件内的具体交互 |
| title | string | 否 | 运营配置 | 事件的中文描述 |
| data1~data7 | string | 否 | 业务定义 | 自定义扩展字段，含义由组件类型决定 |
| data8 | string | 否 | 业务定义 | 通常保留给**人群包 id**，需显式声明 |

## 三、UI 布局驱动的埋点设计规律

通过分析各组件的 UI 截图和对应的 event_name 设计，归纳出以下**视觉布局 → 埋点模式**的核心映射关系：

### 3.1 六种 UI 布局模式及其埋点策略

#### 模式一：Tab 切换布局 → 每个 Tab 独立编号

**典型 UI**：顶部有多个可切换的标签页（如 年卡/双年卡/三年卡、初级场/中级场/高级场、不同时段场次）

**埋点策略**：每个 tab 用 `{前缀}_tab{N}` 或 `{前缀}_{N}` 从左到右编号。

**实例**：
- 支付组件 SKU tab：`topitem_1`(大卡/tab1)、`topitem_2`(小卡/tab2)、`topitem_3`(小卡/tab3) —— 大卡是主推 SKU，小卡是次推
- 抽奖组件场次 tab：`lottery_type_tab1`、`lottery_type_tab2` —— 其中 `type` 替换为 draw/point/hybrid 标识抽奖类型
- 秒杀组件时间段 tab：`session_order_show` —— `order` 替换为 1/2/3 标识场次序号

**为什么这样设计**：Tab 切换意味着用户在不同内容区间做选择，每个 tab 的曝光率和转化率需独立衡量。

#### 模式二：卡片列表/网格布局 → 按位置顺序编号

**典型 UI**：横向排列的商品卡片、纵向排列的任务卡片、网格排列的权益卡片

**埋点策略**：每个卡片位置用 `{前缀}_{N}` 编号，从左到右、从上到下依次递增。

**实例**：
- 权益图 banner（纵向权益介绍列表）：`qyt_1` ~ `qyt_6`
- 底部引导按钮（1~3 个横排按钮）：`button_1`、`button_2`、`button_3`
- 任务列表（双列网格布局）：`task_list_1_name_show` ~ `task_list_6_name_show`，`name` 替换为具体任务名称
- 福利领取组件（横滑卡片）：`equity_show_1`、`equity_show_2`... 从左到右编号

**为什么这样设计**：位置决定了用户注意力分布，编号后可分析不同位置的曝光-点击衰减。

#### 模式三：弹窗/浮层 → _show 展示 + _pop_click 点击配对

**典型 UI**：模态弹窗、半屏弹窗、确认弹窗、结果弹窗

**埋点策略**：弹窗弹出记录展示事件（`xxx_show` 或 `xxx_pop_show`），弹窗内按钮点击记录点击事件（`xxx_pop_click`）。同一组件可能有**多种弹窗状态**，用 `type` 或 `data` 字段区分。

**实例**：
- 权益发放组件有 6 种弹窗形态（主动弹出领取、确认手机号、不满足条件、未绑定手机号、领取成功、领取失败），统一用 `equity_pop_type_show`，`type` 替换为 modal/confirm/reject/unbound/success/fail
- 签到组件弹窗有 6 种类型（当日签到、次日待解锁、次日领取...），用 `sign_window_type_times_show`，data1 区分弹窗类型

**为什么这样设计**：同一弹窗入口可能出现不同结果页面，需要知道用户看到了哪个状态以及后续行为。

#### 模式四：单一 CTA 组件 → 直接语义命名

**典型 UI**：底部悬浮购买条、倒计时条、图片展示、banner 头图

**埋点策略**：只有一个主要展示/交互点时，直接用语义词命名，不加编号。

**实例**：
- 底部下飘支付条（一个价格+一个购买按钮）：展示 `bottomitem_1`，点击 `bottomitem`
- 倒计时条：`countdown`
- 头图 banner：`banner`
- 图片组件：`photo`

**为什么这样设计**：简单组件不需要复杂编号，语义直观即可。

#### 模式五：复合交互流 → 分步事件链

**典型 UI**：礼品卡购买流程（选卡面→选套餐→购买→激活→赠送→接收）、砍价流程（浏览卡片→去砍价→邀人帮砍→直接购买）

**埋点策略**：按用户操作路径拆解为一系列连续事件，每步用 `{组件}_{步骤动作}_click` 命名。

**实例**：
- 礼品卡流程：`gift_card_group_click`(选分组) → `gift_card_cover_click`(选卡面) → `gift_card_sku_click`(选套餐) → `gift_card_btn_click`(购买) → `gift_card_send_btn_click`(赠送) → `gift_card_receiver_btn_click`(接收者操作)
- 亲友卡流程：`friend_card_btn_click`(送好友) → `friend_card_share_pop_click`(选分享方式) → `friend_card_receive_btn_click`(领取者操作) → `friend_card_thank_pop_click`(答谢)

**为什么这样设计**：复杂流程需要追踪每一步的转化率和流失点。

#### 模式六：同组件多样式 → data 字段区分样式

**典型 UI**：支付组件有单卡/横条两种样式；同一组件在不同活动中展示不同布局

**埋点策略**：event_name 保持不变，通过 data1 字段传递样式/模式标识。

**实例**：
- 升级支付组件：event_name 始终为 `upgrade_pay_show_h5`，data1=1(单卡样式) / 2(横条样式)
- 砍价组件：event_name 为 `reduce_price_btn_type_click`，`type` 替换为卡片位置编号

**为什么这样设计**：样式差异不改变事件语义，只是展示形态不同，用 data 字段保持 event_name 的稳定性。

### 3.2 event_name 命名规则速查

| 格式 | 适用场景 | 示例 |
|------|----------|------|
| `{语义}` | 单一展示点 | `banner`, `countdown`, `photo` |
| `{前缀}_{N}` | 多元素位置编号 | `topitem_1`, `qyt_3`, `button_2` |
| `{前缀}_tab{N}` | Tab 切换 | `lottery_type_tab1` |
| `{元素}_show` | 展示事件 | `sku_locked_show`, `bar_show` |
| `{功能}_pop_show` / `_pop_click` | 弹窗展示/点击配对 | `award_pop_show`, `friend_card_pop_click` |
| `{前缀}_type_show` / `_type_click` | 类型替换 | `benefit_type_show`, `equity_pop_type_show` |
| `{前缀}_order_show` | 顺序替换 | `session_order_show` |
| `{功能}_btn_click` | 按钮点击 | `gift_card_btn_click` |
| `{event_name}_{platform}` | 平台后缀 | `upgrade_pay_click_h5` |
| `H5_{event_name}` / `pc_{event_name}` | 平台前缀 | `H5_award_type_show` |

### 3.3 通用规律总结

1. **UI 布局决定编号方式**：Tab 布局用 tab 编号，卡片列表用位置编号，弹窗用状态区分
2. **展示用 show，点击用 click/btn**：展示事件的 event_name 可不带后缀或带 `_show`；点击事件一般带 `_click` 或 `_btn_click`
3. **多个同类元素从 1 开始编号**：从左到右、从上到下，起始为 1
4. **动态占位符用下划线分隔**：`type`、`order`、`name` 等在模板中占位，实际上报时替换为具体值
5. **弹窗必须展示→点击配对**：有 `xxx_pop_show` 就应有 `xxx_pop_click`
6. **跨端复用加平台标识**：H5 端加 `H5_` 前缀或 `_h5` 后缀，PC 端加 `pc_` 前缀或 `_pc` 后缀；全端共用的不加标识
7. **样式差异走 data 字段**：同一组件的不同视觉样式不创建新 event_name，而是通过 data 字段区分

## 四、data1~data8 使用规律

### 4.1 通用语义分层

| 字段 | 常见用途 | 典型取值 |
|------|----------|----------|
| data1 | 主维度：样式类型 / 用户状态 / 操作类型 | `1`(单卡), `2`(横条), `unauth`, `open`, `close` |
| data2 | 次维度：用户购买状态 / 数量 / 次级操作 | `unpurchase`, `purchased`, 数量值 |
| data3 | 业务标识：商品 ID / 弹窗类型码 / 确认状态 | 商品 ID, `gfconfirm`, `lqconfirm` |
| data4 | 业务标识：商品名称 / 确认动作 | 商品名称, `1`(确认) / `0`(取消) |
| data5 | 扩展：月数 / 来源标识 / cycle ID | 月份数, `from_banner` |
| data6 | 扩展：券操作 | `use`(使用) / `cancel`(取消) |
| data7 | 保留扩展 | — |
| data8 | **人群包 id（专用）** | 命中的人群包 id |

### 4.2 常见 data 模式

**用户状态模式**（支付类组件常用）：
```
data1 或 data2 = unauth（未登录）| unpurchase（未购买）| purchased（已购买）| not_matched（不匹配）| ineligible（无资格）| limit（达上限）
```

**确认弹窗模式**（弹窗类组件常用）：
```
data3 = 弹窗类型码（如 gfconfirm, lqconfirm, sfconfirm）
data4 = 1（确认）/ 0（取消）
```

**商品信息模式**（兑换/购买类组件常用）：
```
data3 = 商品ID
data4 = 商品名称
```

**升级支付模式**：
```
data1 = 样式（1=单卡, 2=横条）
data2 = 用户状态
data3 = 升级前的 sku
data4 = 升级目标 sku
data5 = 升级的月数
```

**分享场景模式**：
```
data1 = 分享平台（0=小红书, 1=微博, 2=wps社区）
-- 或 --
data1 = 分享方式（exclusive=专属礼包, group=群发好友, partner=办公搭子）
```

## 五、埋点设计的推理方法

### 核心原则：增量修改，不是从零设计

运营活动的组件是**迭代式发展**的——新 PRD 通常是在已有组件基础上调整功能，而不是全新创建。因此埋点设计也是**在现有基线上增量修改**。

```
Step 0: 加载目标组件的现有埋点基线（ops-activity-baseline.md）
         ↓
Step 1: 从 PRD 中提取业务分析目标
         ↓
Step 2: 对比 PRD 与基线，识别变更点（新增/修改/删除了什么）
         ↓
Step 3: 只针对变更点设计埋点，用 UI 布局确定命名方式
         ↓
Step 4: 输出增量结果（继承 + 新增 + 修改 + 废弃）
```

**如果是全新组件**（基线中无对应组件），则跳过 Step 0 和 Step 2，按完整设计流程走。

### Step 0：加载基线

从 `ops-activity-baseline.md` 中找到目标组件的现有埋点定义，包括：
- 所有 show 事件的 event_name 和 data 字段
- 所有 click 事件的 event_name 和 data 字段
- 平台差异（H5/PC 是否有不同定义）

这是后续所有判断的出发点。

### Step 1：从 PRD 中提取业务分析目标

阅读 PRD 时，关注以下信号词来识别分析需求：

| PRD 中的信号 | 对应的分析目标 | 需要的埋点 |
|-------------|---------------|-----------|
| "对比不同 SKU 的转化效果" | 各 SKU 独立转化率 | 每个 SKU 卡片独立 event_name |
| "提升购买转化" | 漏斗分析：曝光→点击→支付 | 展示+点击完整配对 |
| "测试 A/B 两种样式" | 同一组件不同样式的对比 | 共用 event_name，data 区分样式 |
| "分析用户流失环节" | 多步流程每步转化率 | 每步独立事件，形成事件链 |
| "衡量弹窗挽留效果" | 弹窗触达率 + 弹窗内操作率 | 弹窗展示+弹窗按钮点击配对 |
| "看不同位置/楼层的效果" | 各位置独立 CTR | 按位置编号的 event_name |
| "区分人群包效果" | 不同人群的行为差异 | data8 携带人群包 id |

**如果 PRD 没有显式说明分析目标**，默认追踪以下基础指标：
- 每个组件的曝光量（组件级 show 事件）
- 每个可点击元素的点击率（元素级 click 事件）
- 购买/领取类组件的转化漏斗（展示→点击→结果弹窗）

### Step 2：对比 PRD 与基线，识别变更点

将 PRD 描述的功能与基线对比，标记出每个变更：

| 变更类型 | 判断依据 | 举例 |
|---------|---------|------|
| **继承** | PRD 未提及改动，基线已有 | 支付组件的 `topitem_1/2/3` 未变 |
| **新增** | PRD 中出现的新 UI 元素/交互，基线中不存在 | 新增了第 4 个 Tab、新增了分享弹窗 |
| **修改** | 基线已有，但 PRD 改变了交互逻辑或状态 | 弹窗增加了新的状态类型、data 字段含义变了 |
| **废弃** | 基线有，但 PRD 明确移除了该功能 | 去掉了积分抵现展示 |

**对于"继承"部分**：不需要重新设计，直接引用基线即可。

**对于"新增"和"修改"部分**：进入下面的设计步骤。

### Step 3：确定要追踪的行为和状态

针对变更点，列出具体需要采集的数据：

**行为类**（对应 event_name）：
- 用户看到了什么？→ 展示事件
- 用户点了什么？→ 点击事件
- 用户在哪一步离开了？→ 流程中每步的事件

**状态类**（对应 data 字段）：
- 用户当时的身份状态？→ data 携带 unauth/purchased 等
- 用户看到的是哪种样式？→ data 携带样式标识
- 用户操作的是哪个商品？→ data 携带 sku/商品 ID

### Step 4：判断 event_name 粒度——"拆"还是"合"

| 情况 | 决策 | 原因 |
|------|------|------|
| 分析时需要单独看某个位置的效果 | 拆：每个位置独立 event_name | 位置差异是分析维度 |
| 分析时需要比较不同功能按钮 | 拆：每个按钮独立 event_name | 功能差异是分析维度 |
| 分析时只看整体，不区分状态 | 合：共用 event_name，data 区分 | 状态差异不是主要分析维度 |
| 同一元素在不同样式下表现 | 合：共用 event_name，data 区分 | 样式差异是 A/B 测试变量 |
| 同一弹窗的不同结果 | 合：共用 event_name，data 区分 | 结果分支是同一流程的不同出口 |

**判断口诀**：**位置/功能不同拆 event_name，状态/样式不同拆 data 字段**

### Step 5：用 UI 布局确定命名方式

到这一步才看 UI。前面已经决定了"要埋什么"，UI 布局解决的是"怎么命名和编号"：

| UI 特征 | 命名参考 |
|---------|----------|
| 横向 Tab 切换 | `{前缀}_tab{N}` 或 `{前缀}_{N}` |
| 卡片列表/网格 | `{前缀}_{N}` 从左到右、从上到下编号 |
| 弹窗/浮层 | `{功能}_pop_show` + `{功能}_pop_click` |
| 单一 CTA 按钮 | `{语义}` 或 `{语义}_click` |
| 多步流程 | `{组件}_{步骤}_click` 链式 |
| 轮播/滑动 | `{前缀}_type_show` + 类型替换 |

找不到接近的已有模式时，用组件核心功能词作前缀 + 标准后缀（`_show`/`_click`/`_pop_show`/`_btn_click`）。

### Step 6：规划 data 字段

按语义优先级分配 data1~data8：

1. **data1**：最重要的区分维度（样式类型、操作类型、用户状态）
2. **data2**：次要维度（次级状态、数量）
3. **data3~data5**：业务标识（商品 ID、名称、sku 等）
4. **data6~data7**：低频扩展
5. **data8**：保留给人群包 id

### Step 7：处理平台差异

- **全端通用**：event_name 不加平台标识
- **分端差异化**：加 `_h5` / `_pc` 后缀
- **H5 独有且 PC 已有同名**：加 `H5_` 前缀

### Step 8：输出增量结果

设计完成后，按以下格式输出结果：

**Part 1：继承部分**（列出沿用基线的 event_name，不展开 data 字段细节）

> 以下埋点沿用现有基线，无变更：
> `topitem_1`, `topitem_2`, `topitem_3`, `coupon_1`, ...

**Part 2：变更部分**（重点输出，每条标记类型）

| 类型 | event_name | 变更说明 | data 字段 |
|------|-----------|---------|----------|
| 新增 | `topitem_4` | PRD 新增第 4 个 Tab | — |
| 修改 | `equity_pop_type_show` | type 增加 `expired` 状态 | type 新增值：expired（已过期） |
| 废弃 | `coupon_2` | PRD 移除了积分抵现展示 | — |

### Step 9：自检

设计完成后，回到 Step 1 的分析目标，逐条验证：

- 每个分析目标是否都有对应的埋点数据能支撑？
- 是否存在多余的埋点（没有分析目标对应，但设计了事件）？
- 漏斗链路是否完整（展示→点击→结果，没有断点）？
- data 字段能否区分所有需要对比的维度？

## 六、AI 输出格式

当使用本规范生成埋点方案时，输出以下 JSON 结构：

```json
{
  "component": "目标组件名称",
  "baseline_source": "ops-activity-baseline.md 中的组件章节名",
  "inherited": ["topitem_1", "topitem_2", "topitem_3"],
  "changes": [
    {
      "type": "新增 | 修改 | 废弃",
      "event": "show_rubikwindow | click_rubikbtn",
      "event_name": "事件标识",
      "title": "事件中文描述",
      "position": "楼层编号（由需求定义）",
      "component_name": "PRD 中明确提到的组件名称（如 Banner组件、优惠券组件）",
      "data_fields": {
        "data1": "字段说明",
        "data2": "字段说明"
      },
      "trigger": "触发时机描述",
      "platform": "H5 | PC | 全端",
      "change_reason": "PRD 中的变更原因"
    }
  ]
}
```

**如果是全新组件**（无基线），`inherited` 为空数组，所有埋点都在 `changes` 中以"新增"类型输出。

## 七、关键约束

1. **事件名固定**：只能使用 `show_rubikpage`、`show_rubikwindow`、`click_rubikbtn` 三个事件，不允许新增
2. **event_name 遵循命名规律**：新组件的 event_name 应遵循上述命名模式，保持风格一致
3. **展示与点击配对**：有弹窗展示就应有弹窗点击，有按钮展示就应有按钮点击
4. **编号从 1 开始**：多个同类元素从左到右、从上到下依次编号 1、2、3...
5. **data8 专用**：data8 保留给人群包 id，使用时需显式声明
6. **平台自动字段不设计**：`application`、`url`、`version` 由平台自动上报，方案中无需涉及
7. **增量优先**：已有组件的迭代需求，输出格式必须标注继承/新增/修改/废弃，不能全量重写
8. **基线引用**：修改已有组件时，必须先加载 `ops-activity-baseline.md` 中对应组件的定义
9. **组件名来源约束**：`component_name` 必须来自 PRD 中明确描述的组件名称，严禁编造 PRD 中未提及的组件。PRD 没有提到的组件不需要设计埋点
